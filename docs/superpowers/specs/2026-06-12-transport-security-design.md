# Transport Security (TLS) + Secure Key Storage â€” Design

**Date:** 2026-06-12
**Status:** Approved, ready for implementation plan
**Scope:** Encrypt all Macâ†”phone traffic with TLS and move long-term private keys into platform secure storage (Android Keystore wrapping / macOS Keychain).

## Goal

Today every channel is plaintext: the WebSocket control channel (`ws://`, port 8765), the HTTP file transfer channel (`http://`, port 8766) and the mirror channel (`ws://`, port 8767). Anyone on the same Wi-Fi can read notifications, SMS, clipboard, files and the mirrored screen with a packet sniffer. Independently, the Ed25519 identity keys sit unencrypted on disk (`SharedPreferences` on Android, raw files under `~/Library/Application Support/AirBridge/` on macOS, despite a comment claiming Keychain).

After this change:

- All three channels run over TLS (`wss://` / `https://`) with the Mac's self-signed certificate.
- The phone accepts **only** the certificate whose SHA-256 fingerprint it learned from the pairing QR code (certificate pinning; the QR is the trust anchor, scanned physically).
- The Android Ed25519 private key is encrypted at rest with an Android Keystore master key; the macOS private key and the TLS identity live in the Keychain.

## Non-goals

- No migration path for existing pairings. Both apps are updated together; the user re-pairs once by scanning a fresh QR code. (Decision: accepted by the user 2026-06-12.)
- No certificate rotation mechanism. The certificate is valid ~10 years; rotation = re-pair.
- No hardware-backed Ed25519 on Android (requires API 33+; minSdk is 29). The identity key stays a software key, encrypted at rest.
- No changes to the application protocol. `protocol_version` stays 1 â€” the change is transport-level. An old client simply fails the TCP/TLS handshake against a new Mac.

## Approach (chosen: self-signed TLS + fingerprint pinning)

Alternatives considered and rejected:

- **TLS-PSK** (pre-shared key derived from pairing, no certificates): OkHttp does not support PSK cipher suites; would force replacing the entire Android HTTP/WS stack.
- **Application-layer encryption** (X25519 + AEAD on message payloads): custom crypto, large surface for mistakes, and the streaming HTTP file path would need separate framing. TLS gives the same property with standard, audited code.

## Architecture

### 1. Mac TLS identity (new: `AirbridgeSecurity/TLSIdentityManager`)

On first launch (or when no identity exists) the Mac generates a self-signed certificate:

- EC P-256 key, CN=AirBridge, validity ~3650 days.
- Generated by invoking `/usr/bin/openssl` (LibreSSL ships with macOS; consistent with `scripts/setup-signing-cert.sh`; zero new SwiftPM dependencies). The key/cert pair is bundled into a temporary PKCS#12, imported with `SecPKCS12Import`, and stored as a `SecIdentity` in the **Keychain**. Temporary files are created in a private temp dir and deleted immediately after import.
- Public API: `identity() throws -> SecIdentity` (creates on demand), `certificateFingerprint() throws -> String` (SHA-256 over the certificate DER, lowercase hex).

### 2. TLS on all three listeners (macOS)

`WebSocketServer` (8765 + mirror 8767) and `HttpUploadServer` (8766) switch from `NWParameters.tcp` to TLS parameters built from the same `SecIdentity`:

```swift
let tlsOptions = NWProtocolTLS.Options()
sec_protocol_options_set_local_identity(tlsOptions.securityProtocolOptions,
                                        sec_identity_create(identity)!)
let params = NWParameters(tls: tlsOptions)   // + existing WS options on top
```

Everything above the transport (WebSocket framing, the hand-rolled HTTP parser, mirror binary frames) is unchanged. The parameter construction is injected (the server takes the TLS configuration as a parameter), and integration tests pass a throwaway test identity so they exercise the real TLS handshake (see section 7). Production code always runs TLS; there is no plaintext fallback.

### 3. Trust distribution

- `QRPayload` gains `cert_fingerprint` (snake_case, like the existing fields). The QR remains the only trust anchor.
- The NSD TXT record also carries `cert_fingerprint` â€” **informational only** (lets the phone detect "Mac re-installed, fingerprint changed â†’ needs re-pair" and show a sensible message instead of opaque TLS failures). Trust never comes from NSD.
- Android `PairedDevice` gains `certFingerprint: String`, stored by `PairedDeviceStore` at pairing time.

### 4. Pinned TLS client (Android, new: `network/PinnedTls.kt`)

One shared factory used by all four network clients (`WebSocketClient`, `HttpFileUploader`, `HttpFileDownloader`, `MirrorClient`/`ReverseMirrorClient`):

- A custom `X509TrustManager` that accepts exactly one certificate: SHA-256 over the presented leaf certificate's DER must equal the pinned fingerprint. No CA validation, no expiry check (the pin is the trust statement; self-signed cert).
- `HostnameVerifier` returns true (connections are made to raw IPs; identity is established by the pin, not the name).
- The factory takes the expected fingerprint per connection attempt: `createClient(pinnedFingerprint: String): OkHttpClient` (or a builder-configuring function reused by the existing client classes so their timeout/ping settings stay local).
- URLs change `ws://` â†’ `wss://`, `http://` â†’ `https://`.
- No pin available (device not paired) â†’ connection refused before any socket is opened.

### 5. Secure key storage

**Android (`security/KeyManager.kt`):**

- The Ed25519 private key (PKCS#8, base64) is encrypted with AES-256-GCM using a non-exportable master key generated in `AndroidKeyStore` (`KeyGenParameterSpec`, purpose ENCRYPT/DECRYPT, no user-auth requirement). Stored in the same prefs as `private_key_enc` (base64 of IV + ciphertext); plaintext `private_key_base64` entry is removed.
- One-time migration on first access: if `private_key_base64` exists, encrypt it, store `private_key_enc`, delete the plaintext entry. Key identity (and therefore the public key and pairing) is unchanged.
- Signing API is unchanged; decryption happens inside `KeyManager` when the key is needed.

**macOS (`AirbridgeSecurity/KeyManager.swift`):**

- New `KeychainStorage` implementing the existing `Storage` protocol via `kSecClassGenericPassword` (service `com.airbridge.macos`, `kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly`).
- `Storage.persistent()` returns `KeychainStorage` (making the existing doc comment true). One-time migration: for each known account (`private_key`, `device_id`, `pairedDevices`), if the Keychain has no entry but the legacy file exists â†’ copy into Keychain, delete the file.
- The TLS identity (section 1) lives in the Keychain as a `SecIdentity`, managed by `TLSIdentityManager`.

### 6. Compatibility & failure behavior

- Both apps ship together; there is no mixed-version support window. Old client â†” new Mac fails at the TLS handshake (clean connection error, surfaced as the existing "cannot connect" state).
- After update, the phone has paired devices without `certFingerprint` â†’ treated as not-pairable-over-TLS: NSD auto-connect skips them and the UI directs the user to re-pair (scan QR). The stale entry is replaced on successful re-pairing.
- Wrong certificate presented (fingerprint mismatch) â†’ TLS handshake aborted by the TrustManager, logged with the offending fingerprint, no retry storm (existing ReconnectPolicy backoff applies).
- If the NSD TXT fingerprint differs from the stored pin, the phone shows "Mac certificate changed â€” re-pair required" instead of silently failing handshakes.

### 7. Testing

- **Unit, Android:** TrustManager (matching cert accepted; mismatched rejected; no-pin refuses), AES-GCM wrap/unwrap round-trip + plaintextâ†’encrypted migration, `PairedDevice` (de)serialization with `certFingerprint`, QR payload parsing with the new field.
- **Unit, macOS:** `KeychainStorage` round-trip + fileâ†’Keychain migration (ephemeral test service name), `TLSIdentityManager` creates identity + stable fingerprint, QRPayload encodes `cert_fingerprint`.
- **Integration, macOS:** existing `WebSocketServerTests`/`EndToEndTests` run against the TLS listener with a test identity (test client trusts the test cert). These tests are the real handshake verification.
- **E2E on hardware:** re-pair via QR, clipboard/file transfer both directions, mirror session â€” all over TLS (verify with a packet capture showing TLS, not plaintext, if desired).
